# 用户交互异常轨迹设计文档

> **劳动争议智能咨询系统 · 九步引导式咨询**
> 版本：v0.2
> 日期：2026-04-16
> 状态：初稿

---

## 1. 背景与目的

九步咨询系统采用 LangGraph 状态机驱动，当前交互模式为线性收集：每步按预设问题逐题询问，用户回答后调用 `proceed_to_next_step` 进入下一步。

在实际咨询场景中，用户经常出现**不按引导回答**的情况：
- 跳过关键问题
- 前后回答矛盾
- 使用模糊量词（"还行"、"挺久"）
- 突然切换话题
- 情绪化或表示放弃

本文档定义**五种异常场景下的正确交互轨迹**，作为 LLM prompt 策略的设计规范。
本文档为**基准轨迹 + 五种异常轨迹**的完整集合。

---

## 2. 核心概念

### 2.1 核心事实（STOP_FACTS）

以下五个事实为不可跳过的核心信息，缺失时应阻断进入 `step5_qualification`：

| 编号 | 事实 | 说明 |
|------|------|------|
| F1 | 用人单位名称 | 全称（可描述不记得全称） |
| F2 | 入职时间 | 年份即可，精确到月更好 |
| F3 | 劳动关系状态 | 在职/离职/待岗 |
| F4 | 核心争议内容 | 欠薪/开除/工伤/调岗/社保/其他 |
| F5 | 涉及金额 | 大致数字即可，不需要精确 |

### 2.2 案由分类（6类）

| 案由 | 关键词 |
|------|--------|
| 欠薪 | 工资、加班费、绩效奖金被扣/拖欠 |
| 开除 | 被辞退、解除劳动合同、赔偿金争议 |
| 工伤 | 工作受伤、工伤认定、工伤赔偿 |
| 调岗 | 岗位调整、降薪、工作内容变更 |
| 社保 | 社保欠缴、未缴、断缴 |
| 其他 | 上述以外的其他劳动争议 |

## 4. 异常场景交互轨迹

### 场景 A：用户跳过关键问题

#### 触发条件
用户对核心事实问题（F1-F5 相关问题）直接说"跳过"、"不想说"、"不知道"

#### 交互策略

**Step 1：识别问题性质**

- 判断该问题是否属于核心事实（F1-F5）
- 判断用户是"不愿回答"还是"确实不知道"

**Step 2：分类处理**

```
用户说"跳过"或回避核心问题时：

AI 回应模板：
"理解您可能有顾虑。这个问题（说明为什么重要）是我们计算（赔偿金额/时效）的核心依据。

如果无法给出准确数字，您可以：
A) 告诉我大致范围，我帮您估算
B) 告诉我'不知道'，我通过其他信息帮您判断

如果您暂时不想处理这个问题，我会在后续用[替代数据/标记为待确认]继续。"
```

**Step 3：状态记录**

- 核心事实被标记为 `unknown`
- 若 3 个以上核心事实为 `unknown`，阻止进入 `step5_qualification`
- 显示警告："需要补充至少 3 项核心事实才能生成权益清单"

#### 流程图

```
用户回答"跳过"
    ↓
该问题属于 F1-F5？
    ├─ 否 → 允许跳过，记录为空
    └─ 是 → 进入核心事实处理流程
              ├─ 用户不愿回答 → 解释重要性 + 提供估算/不知道选项
              ├─ 用户确实不知道 → 提供锚点反推（银行流水/社保记录）
              └─ 用户坚持跳过 → 标记 unknown + 累计计数
                                  ↓
                          累计 ≥ 3 个 unknown？
                              ├─ 是 → 阻断进入 step5，显示补充清单
                              └─ 否 → 允许继续，进入下一题
```

### 场景 B：用户回答矛盾

#### 触发条件
用户在当前问题的回答与历史已回答问题存在逻辑矛盾

#### 常见矛盾类型

| 矛盾组合 | 示例 |
|----------|------|
| 就业状态 vs 离职相关描述 | Q1 选择"在职"，Q2 提到"被辞退" |
| 合同状态 vs 工资描述 | 选择"未签合同"，但提到"每月发工资" |
| 时间线矛盾 | 入职 2024年，但提到"干了 5 年" |
| 金额矛盾 | 工资 5000，但赔偿计算 50万 |

#### 交互策略

**Step 1：矛盾检测时机**

- 每次用户回答后，LLM 做一次**交叉验证**
- 检测规则：当前回答与 `step_data` 中已存储的答案进行逻辑比对

**Step 2：矛盾暴露**

```
AI 回应模板：
"我注意到您刚才提到【当前矛盾回答】，但之前您说的是【历史回答】。
这两条信息似乎有些不一致——

请问您的情况是：
A) 【矛盾版本 A】：……
B) 【矛盾版本 B】：……
C) 其他情况（请说明）：……

您可以任选一个，我会据此更新信息。"
```

**Step 3：状态更新**

- 用户确认后，更新 `step_data[stepX].answers`
- 将矛盾涉及的后续步骤加入 `dirty_steps`
- 告知用户："已根据您的最新确认更新信息，相关内容已标记为待刷新"

#### 流程图

```
用户回答
    ↓
LLM 交叉验证（当前回答 vs 历史回答）
    ↓
发现矛盾？
    ├─ 否 → 正常存储，进入下一题
    └─ 是 → 进入矛盾处理流程
              ├─ 展示矛盾点
              ├─ 提供选项让用户选择/澄清
              └─ 用户确认 → 更新 step_data + 标记 dirty_steps
```

### 场景 C：用户回答模糊/不完整

#### 触发条件
用户使用无法转化为结构化数据的模糊量词

| 模糊表达 | 对应结构化需求 |
|----------|----------------|
| "还行"、"差不多" | 月工资金额（number，元/月） |
| "挺久了"、"好几年" | 入职时间（calendar，年/月） |
| "不少"、"挺多" | 涉及金额（number，元） |
| "正常发" | 工资发放方式（radio） |

#### 交互策略

**Step 1：模糊量词识别**

- LLM 在解析用户回答时，检测模糊量词
- 模糊量词包括：还行、差不多、挺久、好几年、不少、挺多、正常、还行吧、大概

**Step 2：追问链**

```
用户回答"工资大概一万左右吧"
AI：[解析为] 月薪 ≈ 10,000 元
    是银行转账还是现金/微信/支付宝？

用户确认发放方式后：
AI：[解析为] 工资发放方式 = 银行转账 ✓
    月薪 = 10,000 元/月 ✓
```

```
用户回答"干了挺久的了"
AI：您记得大概是几年吗？能否告诉我具体的入职年份？
    （如果用户仍无法确定）
    您工资是通过银行卡发放的吗？银行流水可以查到第一笔工资的时间——
    您还记得大概是哪一年入职的吗？
```

**Step 3：锚点反推法**

对于确实无法获得精确值的情况，通过可验证的信息反推：

- 入职时间 → 银行流水第一笔工资记录、社保缴纳起始时间、公积金缴纳记录
- 月工资 → 银行转账记录（标注"工资"项）、个税 APP 收入明细
- 涉及金额 → 欠薪月数 × 月工资

#### 状态记录

- 模糊量词不能直接存入 `answers`
- 必须先转化为结构化数据或标记为 `needs_clarification`
- 提供估算值时，明确标注 `estimated: true`

---

### 场景 D：用户切换话题

#### 触发条件
用户的问题或描述与当前步骤的收集目标无关

#### 话题切换类型

| 类型 | 示例 | 处理策略 |
|------|------|----------|
| 无关咨询 | Q3 时问仲裁申请书格式 | T 型处理：确认→解释→引导回 |
| 抱怨发泄 | 边回答边抱怨公司 | 共情优先，再引导 |
| 追问细节 | 问某个法律条款的含义 | 简短回答 + 继续流程 |
| 主动提供案情 | 没有按顺序，而是开始描述整件事 | 切换到"自由描述"模式 |

#### T 型处理策略（无关咨询）

```
用户（Q3时）：对了，我能问一下仲裁申请书怎么写吗？

AI：[T型处理]
    【确认】您问起了仲裁申请书的问题。
    【解释】这是第8步"文书生成"的内容，在完成全部信息收集后会自动生成。
    【引导】现在我们先继续完成通用问题，这样可以生成更准确的文书。

    Q3. 您签订劳动合同的情况是？
```

#### 共情优先策略（抱怨发泄）

```
用户（Q3时）：我真的受不了他们了，天天让我加班还不给钱……

AI：[情绪检测 → 共情优先]
    您说得我能感受到您的愤怒，这种情况确实让人很委屈。
    您提到的"加班不给钱"——这属于【欠薪】类问题，在后面的权益清单里会帮您列清楚。

    现在先让我了解清楚具体情况，这样之后才能更好地帮您主张权利。
    Q3. 劳动合同签了吗？
```

#### 主动案情描述（切换到自由模式）

```
用户（没有按顺序回答，而是直接开始描述）：
    "我在那个厂子干了三年，前两个月工资一直拖着没发，老板说要倒闭了"

AI：[识别为自由描述模式]
    好的，我已经记录下您说的情况：
    - 工作年限：约3年
    - 问题类型：欠薪（拖发工资）

    您愿意继续这样描述，还是希望我按问题清单逐一询问？
    A) 继续这样描述——您说，我来整理
    B) 按问题清单逐一问——我来问您答
```

#### 流程图

```
用户输入
    ↓
检测话题相关性
    ├─ 相关 → 正常处理
    └─ 不相关 → 判断类型
                  ├─ 无关咨询 → T型处理（确认+解释+引导）
                  ├─ 情绪发泄 → 共情优先（先回应情绪，再处理事情）
                  ├─ 追问细节 → 简短回答 + 继续流程
                  └─ 主动案情 → 切换自由描述模式，提取信息后确认
```

### 场景 E：用户情绪化或放弃

#### 触发条件

用户表达以下意图：
- 放弃："算了"、"不弄了"、"太麻烦了"
- 抵触："我不想回答这些问题"、"太烦了"
- 无助："我不知道该怎么办"、"太复杂了"

#### 情绪/放弃等级

| 等级 | 表现 | 应对策略 |
|------|------|----------|
| 轻度犹豫 | "有点麻烦" | 展示进度 + 简化承诺 |
| 中度抵触 | "太麻烦了不想弄" | 提供保存断点 + 律师求助 |
| 强度放弃 | "算了不要了" | 尊重选择 + 保存进度 + 发送热线 |

#### 分级挽留策略

**轻度犹豫**

```
AI：
我理解您觉得问题有点多，其实很简单：
- 已完成：[3/12] 道题
- 剩下：[9] 道，每次只问一个
- 平均耗时：[3-5] 分钟

后面的问题都是选择题或填数字，很快就能完成。我们继续？
```

**中度抵触**

```
AI：
您可以不用一次性弄完。
- 【保存退出】：我会给您一个链接，随时可以回来继续（凭会话 ID 恢复）
- 【律师求助】：填完基本情况，值班律师帮您处理后续
- 【继续】：剩下的问题我帮您简化

您想怎么做？
```

**强度放弃**

```
AI：
好的，我理解您现在的感受。
您的进度已自动保存（会话 ID：[session_id]），随时可以回来继续。

如果现在需要帮助，可以：
- 拨打工会热线：0527-843XXXXX（工作日 9:00-17:00）
- 前往宿迁市总工会（宿城区太湖路XXX号）

感谢您使用智能咨询系统，祝您顺利。
```

#### 断点保存机制

- 每个步骤完成后自动保存状态到 `step_data`
- 用户可凭 `session_id` 恢复进度
- 恢复时从 `current_step` 指向的步骤继续

#### 流程图

```
用户表达放弃意图
    ↓
判断放弃等级（通过表达强度和关键词）
    ├─ 轻度 → 展示进度 + 简化承诺 + 询问是否继续
    ├─ 中度 → 提供三个选项（继续/保存退出/律师求助）
    └─ 强度 → 尊重选择 + 保存进度 + 发送热线信息 + 结束流程
```

## 5. 状态标记系统

### 4.1 答案状态标记

每个问题的回答存入 `step_data[stepX].answers` 时，附带状态标记：

```python
class AnswerStatus(TypedDict):
    value: Any                    # 回答的值
    status: Literal["confirmed", "fuzzy", "contradictory", "unknown", "skipped"]
    estimated: bool               # 是否为估算值
    needs_clarification: bool   # 是否需要进一步澄清
    clarified_from: Optional[str] # 如果是模糊量词转化，记录原始表达
```

### 4.2 脏数据传播规则

当用户在后续步骤中修正了历史答案时：

```
dirty_steps = get_dirty_range(from_step)
```

- 从修正步骤开始，后续所有步骤标记为 dirty
- 用户进入被标记的 dirty 步骤时，LLM 提示："检测到之前的回答已被修改，当前内容已标记为待更新"

## 7. 实现建议

### 6.1 LLM Prompt 增强点

在现有 `STEP_PROMPTS` 的基础上，每个步骤的 prompt 增加：

1. **异常场景处理指令**：描述当前步骤可能出现的异常及处理策略
2. **矛盾检测规则**：列出与该步骤问题相关的历史矛盾检测规则
3. **核心事实检查点**：说明该步骤涉及哪些 F1-F5，如何处理缺失
4. **情绪关键词库**：该步骤可能出现的情绪表达及应对话术

### 6.2 关键工具

| 工具 | 用途 | 调用时机 |
|------|------|----------|
| `request_missing_info` | 追问缺失/模糊信息 | 用户回答无法结构化时 |
| `proceed_to_next_step` | 进入下一步 | 必要数据已获取 |
| `back_to_previous_step` | 返回修改 | 用户发现之前回答有误 |
| 新增：`clarify_contradiction` | 暴露矛盾请求澄清 | 检测到矛盾时 |
| 新增：`detect_emotion` | 情绪检测（非工具，返回信号给 node） | 每轮对话 |

### 6.3 验证机制

每个步骤结束前，LLM 验证：
- 必要字段是否完整（核心事实不能全为 unknown）
- 是否有未解决的矛盾标记
- 是否有未处理的情绪信号

不满足则不调用 `proceed_to_next_step`，持续追问。
