# 从自由问答到九步引导式智能咨询系统：宿迁工会劳动争议咨询系统的演进之路

## 前言

2026年初春，我们接到了一个特殊的项目：为宿迁总工会开发一套劳动争议智能咨询系统。项目的目标看似简单——让一位完全不懂法的普通职工，通过与AI助手的对话，就能清晰地了解自己的权益、知道下一步该怎么做、最终顺利完成维权。但真正开始动手时，我们才发现这个"简单"目标背后，藏着多少意想不到的挑战。

本文将完整记录这个项目从诞生到实现的全过程：技术选型的纠结、架构演进的阵痛、与大模型反复"搏斗"的经验教训，以及最终沉淀下来的方法论。这不是一篇技术流水账，而是一段关于如何在真实业务场景中用好AI的实践思考。

---

## 一、从痛点出发：为什么要做九步引导式咨询

### 1.1 自由问答的根本困境

项目最初，我们参照大多数法律AI的思路，设计了一套"自由问答"系统：用户描述自己的情况，AI给出解答和建议。这套方案在Demo阶段效果不错——用户输入"我被公司辞退了怎么办"，AI能洋洋洒洒写出一篇法律分析。

但当我们真正面向目标用户——工厂工人、外卖骑手、滴滴司机——测试时，问题出现了。

一位大姐输入"公司不要我了"，AI回复了三千字的法律条文。大姐打电话来问："你说的这些我都知道，我就是不知道该怎么办，你能不能直接告诉我下一步做什么？"

这让我们意识到一个根本性的问题：**法律咨询不是信息战，而是决策战**。普通劳动者不需要知道《劳动合同法》第十四条怎么写，他们只需要知道："你现在可以打12333投诉，也可以申请劳动仲裁，以下是具体步骤"。他们需要的是一张清晰的路线图，而不是一本法律百科全书。

### 1.2 从"你问我答"到"我带你走"

基于这个洞察，我们提出了"九步引导式咨询"的概念。核心思路是：**不再让用户自己组织问题，而是系统主动一步步引导用户完成维权的全流程**。

这九步是：

1. **模式选择** — 找律师还是AI咨询？
2. **问题初判** — 自述案情还是引导问答？
3. **信息补全** — 通用问题 + 个性问题
4. **案件定性** — AI自动形成案件事实描述和案由判定
5. **证据攻略** — 该准备什么证据？已有多少？
6. **风险提示** — 维权路上有哪些坑？
7. **文书生成** — 自动生成仲裁申请书
8. **行动路线图** — 下一步具体怎么走
9. **求助复核** — AI复核 + 一键求助工会律师

这九步的设计参考了真实劳动仲裁的流程，同时也考虑了普通用户的认知负担。每一步都有明确的目标和交付物，用户不需要理解法律体系，只需要跟着系统走，就能完成从"遇到问题"到"知道怎么办"的转变。

## 二、技术架构：从"多Agent协作"到"九步状态机"

### 2.1 最初的Supervisor架构

项目启动之初，我们参考业界流行的多Agent架构，设计了一套"Supervisor Architecture"：

- **协调者（Supervisor）**：统一接收用户问题，决定由哪个专业角色处理
- **解答者（Answerer）**：负责初步法律分析
- **情形分析员（Detailer）**：推演不同情形下的维权走向
- **路径规划师（Advisor）**：规划具体维权步骤
- **文书助手**：协助撰写法律文书

这套架构在技术上演示效果不错，但很快暴露出了问题：

**问题一：职责边界模糊**。当用户说"公司让我走人"，是让解答者先分析，还是让路径规划师直接规划？协调者的判断逻辑越来越复杂，最后变成了另一个"硬编码的工作流"。

**问题二：多Agent通信开销大**。每次用户输入都要经过协调者分发、专业角色处理、结果返回，响应延迟明显。更重要的是，多轮对话后各角色的上下文累积导致token消耗剧增。

**问题三：输出不可控**。不同角色生成的内容风格不一，有时专业术语太多，有时又过于口语化。用户拿到的是一盘"大杂烩"。

### 2.2 LangGraph状态机：让流程"固定"下来

经过反复讨论，我们决定放弃多Agent协作架构，改用**LangGraph状态机**来实现九步流程。

LangGraph是LangChain团队推出的一个用于构建有状态、多步骤AI应用的框架。它的核心概念是**StateGraph**——一个由节点（Nodes）和边（Edges）组成的有向图，每个节点代表一个处理步骤，每条边代表状态转换的规则。

关键的是，每个节点内部可以是**一个大模型调用**，也可以是**一个包含大模型调用的子图**。这种嵌套结构非常适合我们的九步流程：主图负责步骤间的路由，每个步骤内部由各自的专业prompt驱动。

我们最终采用的架构是这样的：

```
主StateGraph（固定骨架）
    │
    ├── step2_initial（问题初判）
    │       └── create_agent + prompt
    ├── step3_common（通用12问）
    │       └── create_agent + prompt
    ├── step4_special（特殊问题）
    │       └── create_agent + prompt
    ├── step5_qualification（案件定性）
    │       └── create_agent + prompt
    ├── step6_evidence（证据攻略）
    │       └── create_agent + prompt
    ├── step7_risk（风险提示）
    │       └── create_agent + prompt
    ├── step8_documents（文书生成）
    │       └── create_agent + prompt
    ├── step9_roadmap（行动路线图）
    │       └── create_agent + prompt
    └── step10_review（求助复核）
            └── create_agent + prompt
```

**核心设计决策**：每个步骤是一个独立的`create_agent`子图，通过`Command(goto=..., graph=PARENT)`在步骤间跳转。这种设计的优势是：

1. **步骤边界清晰**：每个步骤只负责自己的业务逻辑，不关心其他步骤
2. **状态集中管理**：主图维护全局状态`ConsultationState`，包括`current_step`、`case_category`、`evidence_items`等
3. **回退机制自然**：从step8可以跳回step3修改，dirty标记自动传播

### 2.3 前端-后端接口约定

在九步状态机的设计中，有一个关键决策：**前端感知步骤，后端控制流转**。

具体来说：

- 前端通过URL和UI状态感知用户"在哪里"（step3还是step8）
- 但实际的步骤推进由后端状态机决定
- 前端发送用户输入 → 后端判断是否满足跳转条件 → 决定继续当前步骤还是跳转下一步
- 跳转时，后端通过SSE（Server-Sent Events）推送`current_step`更新，前端据此更新UI

这个设计的好处是：**业务逻辑（什么时候该问什么问题）全部在后端，前端只负责渲染**。前端不需要知道"欠薪案件step4应该问哪些特殊问题"，只需要把用户的回答传过去，后端会自己处理。

## 三、实战经验：大模型应用开发的"坑"与"道"

### 3.1 LangGraph的那些坑

开发九步状态机的过程中，我们踩过太多LangGraph的坑。以下是几个最典型的：

**坑一：Command.PARENT必须显式指定**

在子图内部使用导航工具时，必须明确指定`graph=Command.PARENT`，否则跳转只在子图内部生效，父图完全"看不到"这次跳转。

```python
# 正确
return Command(goto="step5_qualification", update=updates, graph=Command.PARENT)

# 错误：缺少graph=Command.PARENT
return Command(goto="step5_qualification", update=updates)
```

这个错误我们在调试时犯过很多次——子图明明执行完了，状态也更新了，但就是不跳转到下一步。排查半天才发现是缺少了`graph=Command.PARENT`。

**坑二：导航工具必须同时更新current_step和active_agent**

只更新`active_agent`而遗漏`current_step`，会导致步骤计数器停滞，下次跳转时父图的判断逻辑就会混乱。

```python
# 正确：两个字段都要更新
updates = {
    "active_agent": "step5_qualification",
    "current_step": 5,
}
```

**坑三：消息历史要用messages_add而不是_replace_list**

`messages`字段在状态中是累积的，每轮对话新增的AI消息和用户消息应该追加到历史中，而不是替换整个历史。如果用`_replace_list`，会导致消息丢失。

```python
from operator import add as messages_add

messages: Annotated[List[AnyMessage], messages_add]  # 正确：累积
```

**坑四：子图内无法直接访问父状态**

`create_agent`创建的子图是状态隔离的。在子图内调用`runtime.state`拿到的是子图自己的状态，而不是父图的状态。如果需要在子图内访问父状态的数据，必须通过`update`参数传递。

### 3.2 SSE流式输出的教训

九步流程的内容生成靠的是流式输出——后端通过SSE（Server-Sent Events）实时推送AI生成的内容，前端逐字渲染并显示闪烁光标。这听起来简单，实际开发中却踩了不少坑。

**教训一：不要修改消息内容的换行符**

SSE协议本身用`\n\n`分隔不同事件。如果对content中的换行符做替换，会破坏多行消息的格式。正确做法是直接`json.dumps`，JSON会正确转义换行符。

**教训二：LLM的chunk超时要设长**

最初我们将SSE的chunk超时设为30秒。但LLM推理有时需要较长时间，30秒太短，导致大量请求超时失败。最终我们将超时调整为120秒。

**教训三：必须过滤消息类型**

SSE推送时，前端只需要接收`ai`类型的消息。`system`、`human`、`tool`类型的消息应该过滤掉，否则前端会渲染出乱七八糟的内容。

```python
if msg_type not in ("system", "SystemMessage", "human", "tool"):
    yield f"event: content\ndata: {...}\n\n"
```

### 3.3 前端状态管理的教训

前端开发中，最让我们头疼的是**前端状态和后端状态不同步**的问题。

**问题一：Session ID用占位符**

最初我们用字符串`'new'`作为新会话的session_id。这导致多个浏览器标签页共享同一个会话——在标签A对话，标签B也在对话，状态完全混乱。

解决方案：前端生成真实的UUID v4作为session_id。

```typescript
function generateUUID() {
  return 'xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx'.replace(/[xy]/g, function(c) {
    const r = Math.random() * 16 | 0
    const v = c === 'x' ? r : (r & 0x3 | 0x8)
    return v.toString(16)
  })
}
```

**问题二：流式状态和消息状态混在一起**

最初我们用同一个state管理所有聊天消息和流式内容。但流式过程中频繁更新state，导致race condition——某些消息还没完全接收就被覆盖了。

解决方案：流式过程中用独立的`streamingContent`追踪，完成后才更新`messages`数组。

**问题三：错误静默失败**

最初有些catch块是空的`catch {}`，所有错误都被吞掉，调试时完全不知道哪里出了问题。

解决方案：catch块必须至少`console.error`记录错误，并通过error state渲染到UI。

## 四、AI辅助开发的方法论

### 4.1 两种AI，两个层次

这个项目让我们深刻认识到"AI"在开发中的双重角色：

**第一层：AI作为开发工具（AI Coding）**。我们使用Claude Code辅助写代码、重构、调试。这是大多数开发者熟悉的用法。

**第二层：AI作为应用的核心组件（AI Application）**。我们构建的系统本身以LLM为核心能力——九步咨询流程的每一步都是一个大模型在驱动内容生成。

有趣的是，这两个层次的"AI调试"逻辑完全不同：

- **AI Coding的错误**：通常是传统软件工程的问题——类型错误、逻辑错误、边界条件缺失。用传统调试方法就能解决。
- **AI Application的错误**：往往是LLM行为不可预测导致的——prompt怎么写都不对、LLM"自作主张"替用户做选择、无限循环不推进流程。这种错误没有固定的调试套路，只能通过反复测试-反馈-修复来逼近理想行为。

### 4.2 测试驱动 vs. AI自检

传统开发中，我们习惯于"AI能自我检查"——写完代码，AI能自己发现bug。但在LLM应用开发中，这个假设经常失效。

一个典型的例子：step4_special的prompt，让AI根据案由类型生成特殊问题。我们让AI自己检查输出的问题是否符合案由逻辑，AI满口答应"没有问题"。但实际跑一遍流程就发现：无论什么案由，AI都在问同样的通用问题，根本没有根据案由类型做筛选。

后来我们总结出一个原则：**开发者是测试的执行者，AI是修复的执行者**。不要依赖AI的自我判断，而是要开发者主动设计测试用例，引导AI走完完整流程，发现偏差后准确描述给AI，再让AI修复。

### 4.3 禁止规则比描述性规则更有效

在给所开发的LLM应用编写提示词时，我们发现了一个反直觉的规律：**正向描述不如负向禁止有效**。

LLM有强烈的"生成内容"倾向。当你写"应该用文字展示选项"，LLM可能听；但当你写"禁止替用户做选择"时，LLM更容易遵守。

具体来说：当我们在提示词中描述期望行为（如"如果用户没有明确选择，应该用文字展示选项供用户选择"），LLM仍然可能自作主张地调用交互工具替用户做选择。但当我们改用禁止语法（如"禁止替用户做选择，禁止在用户未明确选择时调用交互工具"），问题消失了。

原因是：AI在训练中被大量"生成内容"的样本训练过，"生成"的冲动往往强于"遵守约束"的冲动。显式的禁止规则能更好地校正这种倾向。

更深层的规律是：**LLM更倾向于"做"什么，而不是"等"什么**。当你描述一种期望行为时，LLM会尝试去实现这个描述；但当你明确禁止某种行为时，LLM会主动避开这个禁区。这和我们在日常对话中习惯于"说目标"相悖——对LLM应用写提示词时，约束往往比目标更有效。

### 4.4 边界条件需要开发者主动触发

LLM更关注正常路径，容易忽略边界情况。这是训练数据中正常路径出现频率远高于异常路径造成的。

我们在开发中养成了一个习惯：每次AI完成主流程后，主动问自己——"如果这个值为空会怎样？""如果用户第一步就放弃会怎样？""如果网络中断会怎样？"——然后把这些问题抛给AI，让它补充边界处理逻辑。

### 4.5 外部环境问题AI无法自愈

当网络波动、API限流、第三方服务异常时，AI会反复重试同一操作并报错，但不会想到检查环境配置。这种时候必须开发者介入，而不是让AI继续试错。

## 五、架构决策的思考

### 5.1 自由度与可控性的平衡

完全依赖大模型：灵活但输出不可控，容易"一本正经胡说八道"。
完全硬编码工作流：稳定但缺乏灵活性，无法应对法律咨询中千变万化的情形。

我们的最终选择是**"约束下的自主推理"**：通过LangGraph状态机定义好各步骤的边界和跳转规则，确保流程不会失控；通过精心设计的prompt为每个步骤设定专业的思维框架，使其在自己擅长的领域内自主推理。

换句话说，AI在受约束的范围内自由思考——既要发挥大模型的推理能力，又要保证输出在可控可预期的轨道上。

### 5.2 复杂度控制是开发者的责任

项目开发中有一个让我们印象深刻的教训：前端曾尝试引入"Generative UI"（选项卡片、动态表单等前端交互组件），让AI直接生成可交互的UI组件。

这个想法很美好——AI可以根据当前步骤生成对应的输入组件。但实际开发中发现，这大大增加了系统的复杂性：前端需要解析AI生成的JSON schema来渲染组件，需要处理各种异常情况，整个系统的复杂度超出了我们的处理能力。

最终我们主动踩了刹车：**放弃Generative UI，让AI仅通过纯对话与用户交流**。输入方式虽然变成了纯文本，但系统的稳定性和可维护性大幅提升。

这个决策告诉我们：**AI不擅长判断系统复杂度的边界**。当项目复杂度超出AI处理能力时，AI不会主动说"太复杂了，先做简单的"。它会继续尝试解决当前问题，直到开发者喊停。复杂度控制、对系统边界的把控，必须由开发者来负责。

### 5.3 前端-后端职责划分

项目中有过一个反复讨论的设计问题：**交互工具（如select_option）放在前端还是后端？**

最开始的方案是：AI需要用户选择时，生成一个JSON描述前端应该渲染什么组件（如下拉菜单、单选按钮等）。前端解析这个JSON，渲染对应的组件，用户操作后返回结果给后端。

这个方案的问题是：前端需要理解AI的"意图"，解析JSON schema，处理各种边界情况。复杂性急剧上升。

最终我们采用了**后端完全控制**的方案：AI通过`select_option`等tool与用户交互，后端管理所有状态，前端只负责渲染AI返回的文本内容和显示/隐藏预设的表单组件。

这个决策让前端的工作变得简单——它不需要理解AI在想什么，只需要把后端推送的内容渲染出来即可。

## 六、项目沉淀

### 6.1 核心成果

经过约两个月（实际开发时间约三周）的迭代，项目最终交付了一个可用的九步引导式劳动争议咨询系统：

- **后端**：FastAPI + LangGraph九步状态机，支持SSE流式输出、会话持久化（InMemorySaver）、断点恢复
- **前端**：React + TypeScript + Tailwind CSS，通过SSE实时接收AI输出，整合了ChatPage聊天界面和InfoPanel信息面板
- **数据**：6大高频案由分类（欠薪/开除/工伤/调岗/社保/其他）+ 三级案由体系 + 证据清单数据库
- **文档**：完整的需求文档、技术方案、工作日志、AI开发经验总结

### 6.2 效率提升的量化

相比传统开发方式，AI辅助开发在以下环节带来了显著效率提升：

| 环节 | 传统方式 | AI辅助 | 提升幅度 |
|------|---------|--------|---------|
| 架构方案枚举 | 3天 | 2小时 | ~10x |
| Prompt迭代优化 | 1周 | 1天 | ~5x |
| 需求差距扫描（29项） | 2天 | 2小时 | ~8x |
| 多文件一致性修改 | 手动，容易遗漏 | AI批量处理 | 质量提升 |

更重要的是，AI辅助让我们能在短时间内完成更多轮的"生成-测试-修复"循环。人工做这件事会疲惫，AI不会。这直接提升了最终交付质量的上限。

### 6.3 沉淀的方法论

项目过程中，我们沉淀了以下可复用的方法论：

**测试驱动的迭代优化**：开发者是测试的执行者，AI是修复的执行者。不要依赖AI的自我判断，通过实际运行来验证是否符合预期。

**禁止规则与状态变量并用**：要让AI遵守行为约束，不能只依赖正向描述（"应该做什么"），还需要负向约束（"禁止做什么"）。

**上下文锚定法**：AI的输出质量高度依赖输入的上下文。提供的数据文件越具体、案例越丰富，AI的输出质量越高。

**增量优先于全量**：AI擅长对某类变更进行增量修改，不擅长全盘重写。每次只下达一种类型的变更指令，发现问题可以快速定位和回滚。

**环境异常的人工介入原则**：当AI反复重试同一操作并持续失败时，应该立即考虑开发者介入。AI不具备检查环境配置的能力。

## 七、写在最后

这个项目给我们最深的体会是：**AI是强大的工具，但不是万能的解决方案**。

在正确的场景下，AI能大幅提升开发效率——枚举方案、生成初版、迭代优化——这些它都做得很好。但在边界条件判断、复杂度控制、架构决策这些需要"工程判断力"的地方，AI的表现往往不如人类。

更重要的是，**用AI开发AI应用，比用AI开发传统软件更难**。因为prompt和代码是交织在一起的，修改一处可能影响另一处，而这种跨层关联恰恰是AI不擅长的领域。

对于未来，我们认为最可行的路径是：**让AI做它擅长的事（生成、枚举、迭代），让人类做AI不擅长的事（判断、决策、控制）**。在两者的边界处，人类要保持清醒——知道什么时候该信任AI，什么时候该介入纠正。

最后，这个项目的成功离不开宿迁总工会的业务支持，也离不开Claude Code和DeepSeek模型的强大能力。技术从来不是孤立的，它始终服务于真实的业务需求和真实的用户。
